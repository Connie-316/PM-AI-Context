## 一、需求池说明
 
```
需求池 = 所有进来的需求的"蓄水池"

规则：
1. 任何来源的需求都要登记（组长/领导/老板/客户反馈）
2. PM 评估优先级，不直接承诺时间
3. 已评审的需求才进入开发计划
4. 变更的需求要记录变更原因
```

## 三、需求优先级定义

| 优先级 | 含义 | 处理规则 |
|--------|------|---------|
| P0 | 紧急，影响营收/已上线重大bug | 立即处理，当天响应 |
| P1 | 高，本月上线计划相关 | 本周评审，排入当前迭代 |
| P2 | 中，提升效率/体验优化 | 下月评审，排入下月迭代 |
| P3 | 低，想法很好但可缓 | 排期待定，不承诺时间 |

## 六、使用说明

### 6.1 收到新需求时

```
1. 打开本文档
2. 在"待评审"区新增一行
3. 填写：需求名称/来源/提出时间/优先级（P3初始）
4. 找来源方确认背景和目标
5. 评估后更新优先级
6. 通知来源方预计评审时间
```

### 6.2 需求状态流转

```
待评审 → 评审通过 → 开发中 → 测试中 → 已上线 → 已归档
   ↓         ↓
  拒绝     评审不通过 → 已取消
```

### 6.3 需求变更时

```
1. 在"需求变更记录"区新增一行
2. 记录：原需求/变更内容/变更原因/时间
3. 通知相关人（研发/测试）
4. 更新TAPD任务状态
5. 更新PRD（如已写）
```
