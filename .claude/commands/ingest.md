# 知识采集 Skill（Ingest）

## 使用场景

当用户说"记录这个"、"保存这个"、"加到知识库"、"这个信息存一下"时调用。
用户粘贴原始文本、语音转文字、手动输入的内容，都通过这个 Skill 路由到正确位置。

---

## 核心原则

**内容分类决定存放位置，不要凭感觉路由。**

```
用户输入 → 分类判断 → 路由到对应目录 → 格式化 → 更新索引 → 确认
```

---

## 内容分类与路由

| 用户说 | 内容类型 | 目标目录 | 文件决策 |
|--------|---------|---------|---------|
| "idea"、"也许可以"、"what if" | 探索 | drafts/ | 新建文件 |
| "meeting"、"我们讨论了"、"结论是" | 会议记录 | workspace/ | 新建或追加 |
| "requirement"、"要做一个"、"这个功能" | 需求 | workspace/需求池管理.md | 追加到需求池 |
| "这是我们产品的逻辑"、"标准流程是" | 已确认事实 | context/对应目录 | 新建或追加 |
| "竞品XX做了"、"市场数据" | 参考资料 | context/01_产品架构/ | 新建文件 |
| 无法判断 | 默认 | drafts/ | 新建文件 |

**原则：不确定就放 drafts/，晋升比降级容易。**

---

## 执行步骤

### Step 1：分类判断

读取用户输入，判断属于哪种类型。无法判断时默认 drafts/。

### Step 2：目录路由

```
探索（Exploration）        → drafts/
会议记录（Meeting）        → workspace/ + 命名加日期前缀
需求（Requirement）        → workspace/需求池管理.md
已确认事实（Fact）         → context/对应目录
参考资料（Reference）       → context/01_产品架构/
默认（Unclear）            → drafts/
```

### Step 3：文件决策

**新建还是追加？**

| 场景 | 决策 |
|------|------|
| 命名包含日期（如会议记录） | 每次新建 |
| 同类内容持续积累（如需求池） | 追加到现有文件 |
| 新主题首次出现 | 新建文件 |
| 已有类似主题 | 追加到现有文件 |

**追加规则：**
- 需求 → 追加到 workspace/需求池管理.md 的"待评审"区
- 会议记录 → 新建文件，不追加
- 术语/事实 → 检查是否已有，已有则追加解释段落

### Step 4：格式化写入

使用对应模板（见下方），或直接按 markdown 格式写入。

### Step 5：更新索引

文件创建后，必须更新目标目录的 CLAUDE.md 索引。

### Step 6：确认用户

回复格式：
```
✅ 已保存到：[目录/文件]

**内容摘要：**
[3行以内]

**分类判断：**[类型] → [目录]
**如需修改，告诉我"移到 XXX"或"改一下内容"**
```

## 分类模板

### 探索（drafts/）

```markdown
---
description: [一句话描述]
type: exploration
created: [日期]
---

# [探索标题]

## 背景
[为什么有这个想法]

## 核心想法
[具体描述]

## 待验证
- [ ] 这个想法是否值得推进？
- [ ] 有没有类似实现？
- [ ] 风险点是什么？
```

### 会议记录（workspace/）

```markdown
# 会议纪要：[主题]

## 会议信息
- 日期：[YYYY-MM-DD]
- 参与人：[名单]
- 主持人：[名字]
- 记录人：[名字]

## 议程
1. [议题1]
2. [议题2]

## 讨论内容
### 议题1
- [结论]
- [待确认事项]

## 行动项

| 行动 | 负责人 | 完成时间 |
|------|--------|---------|
| [行动] | [@人] | [日期] |

## 下次会议
- 时间：[日期]
- 议题：[议题]
```

### 需求（workspace/需求池管理.md）

追加到"待评审"区：
```
| [ID] | [需求名称] | [来源] | [YYYY-MM-DD] | [P3初始] | 待评审 | [备注] |
```

### 已确认事实（context/对应目录）

```markdown
# [事实标题]

## 定义
[一句话精确定义]

## 详细说明
[2-3句展开]

## 适用范围
- 在 [场景A] 使用
- 在 [场景B] 不适用

## 相关
- 关联：[术语/文档名称]
```

### 参考资料（context/01_产品架构/）

```markdown
# [资料标题]

## 来源
[出处/时间/可信度]

## 核心内容
[3-5句概括]

## 关键数据
| 指标 | 数值 |
|------|------|
| [指标1] | [数值] |
| [指标2] | [数值] |

## 对我们的参考意义
[1-2句]

## 原文链接
[链接]
```

## 自动触发

**以下情况自动调用 Ingest，不需要用户说：**
1. 用户粘贴超过 200 字的内容
2. 用户发送截图并说"这是什么"、"帮我分析"
3. 用户说"背景是"、"我们的业务是"

---

## 与其他 Skill 的配合

```
用户粘贴内容 → Ingest 分类路由
    ↓
如果是要写 PRD → prd-write 接手
    ↓
如果是要做会议记录 → 进入 workspace/
    ↓
完成后 → 更新索引
```

---

## 边界判断

| 情况 | 判断 | 路由 |
|------|------|------|
| 用户说"把这个需求记下来" | Requirement | workspace/需求池管理.md |
| 用户说"把竞品分析记下来" | Reference | context/01_产品架构/竞品分析.md |
| 用户说"这个想法先放着" | Exploration | drafts/ |
| 用户说"这是我们的标准流程" | Fact | context/对应目录 |
| 用户只说一句话需求 | Requirement | workspace/需求池管理.md（追加） |
| 用户粘贴了一段 PRD 文字 | Fact（PRD内容） | context/对应目录 + 更新 workspace/prd/ |
