# TAPD 迭代对齐 Skill（tapd-sync）

## 使用场景

当你说"TAPD对齐"、"迭代版本乱了"、"需求被跳过"、"组长直接写了需求"时调用。
这是专门针对你实际工作场景的实操指南。

---

## 你的实际问题

```
问题1：组长把迭代版本去掉了（序列号消失）
问题2：有些需求没经过你处理但要在迭代版本里写
问题3：开发取消迭代版本 → 看不到那个需求了
问题4：上线迭代的需求要单独显示
问题5：图三红色那些不知道是谁写的（组长的工作）
```

---

## 核心原则

```
1. TAPD 是"执行记录"，不是"规划记录"
2. 你需要自己的"规划记录"来对照TAPD
3. 被跳过/取消的需求 → 你的规划文档里标记状态
4. 组长直接写的需求 → 你的规划文档里也记录（来源=组长）
5. 上线迭代需求单独列 → 和TAPD的"已完成"对应
```

---

## 执行步骤

### Step 1：建立自己的迭代规划表（每月）

这个表存在 workspace/ 里，**不是TAPD**，是你自己的规划对照表。

```markdown
# [月] 迭代规划对照表

## 本月迭代规划（TAPD对照用）

| 序号 | 任务名称 | TAPD编号 | 迭代版本 | 来源 | 状态 | 备注 |
|------|---------|---------|---------|------|------|------|
| 1 | 合同管理 | [编号] | v1467 | 组长 | ⏳进行中 | |
| 2 | 在线询价 | [编号] | v1466 | 组长 | ✅已上线 | |
| 3 | 订单管理 | [编号] | v1465 | 你 | 🔴已取消 | 开发说做不了 |
| 4 | XXX需求 | - | 被删除 | 组长 | ⚠️未进入迭代 | 直接给开发了 |

## 来源说明
- 我 = 你创建的
- 组长 = 组长直接写/修改的
- 领导/老板 = 其他领导插队

## 状态说明
- ⏳进行中：在当前迭代
- ✅已上线：在上线迭代里
- 🔴已取消：被开发/组长取消
- ⚠️未进入迭代：没在TAPD里但实际在做
```

### Step 2：每周同步TAPD到规划表

```markdown
# [日期] TAPD同步记录

## 本周TAPD变化
| 变化类型 | 任务 | 原迭代 | 新迭代 | 变化原因 |
|---------|------|-------|-------|---------|
| 删除 | XXX | v1464 | - | 组长去掉了 |
| 新增 | YYY | - | v1468 | 组长直接写 |
| 取消 | ZZZ | v1463 | 🔴取消 | 开发说做不了 |

## 红标任务（上线迭代）
| 任务 | 来源 | 实际上线日期 | 备注 |
|------|------|------------|------|
| 合同管理 | 组长 | 2026-05-15 | 上线了但没经过我 |
```

### Step 3：遇到"被跳过"的需求时

```
场景：组长直接找开发做了个需求，没经过你，TAPD里也没有

处理：
1. 在规划表里记：任务名/来源=组长/状态=⚠️未进入迭代
2. 不需要追（组长的工作不用追）
3. 只需要知道"有这件事"，方便写迭代版本总结
```

### Step 4：TAPD迭代版本被删的处理

```
场景：组长把某个迭代版本从TAPD里删掉了，但实际需求还在做

处理：
1. 在规划表里记：原迭代版本 → 状态=🔴已取消/⚠️未进入迭代
2. 追问原因："这个需求取消了吗？还是移到其他迭代了？"
3. 如果移到其他迭代 → 更新迭代版本
4. 如果真的取消 → 标记取消原因
```

## 实际操作模板

### 场景A：每周一早上同步TAPD

```
1. 打开TAPD，看当前迭代版本列表
2. 打开 workspace/[月]迭代规划对照表.md
3. 对照：
   - TAPD里有 → 规划表里有没有？没有就加上
   - TAPD里没有（但实际在做）→ 规划表里标⚠️
   - 规划表里有 → TAPD里有没有？没有就问"这个还做吗"
4. 更新规划表
5. 早会汇报时用规划表（不用报TAPD序列号，说任务名）
```

### 场景B：组长又直接找开发了

```
不要追，不要质疑。

1. 在规划表新增一行：
   | 新需求 | - | - | 组长 | ⚠️未进入迭代 | 直接给开发了 |
2. 后续：如果上线了，标记"✅已上线"
3. 如果没上线，标记"🔴取消"
4. 月底写迭代总结时，这些也要写进去（来源=组长）
```

### 场景C：开发说"这个迭代取消"

```
1. 问清楚原因（记录到规划表）
2. 在规划表更新：该任务状态=🔴已取消
3. 月底复盘时统计：本月取消X个，原因是XX
4. 如果是PRD的问题 → 反思PRD是否没写清楚
5. 如果是需求变更 → 记录变更原因
```

## 与需求池管理的关系

```
需求池管理.md = 所有需求进来时的记录（来源/优先级/状态）
迭代规划对照表 = 实际TAPD迭代版本的对照（执行层面的记录）

两个一起用：
1. 新需求 → 先记需求池管理.md
2. 进入迭代 → 同步到迭代规划对照表
3. 迭代变化 → 更新迭代规划对照表
4. 上线 → 两个都标记已完成
```

---

## 输出格式

每次同步后输出：
1. 更新后的迭代规划对照表
2. 本周变化摘要（2-3句话）
3. 早会汇报素材（任务名+状态，不报TAPD编号）

```
每周一：早会前同步一次
每月25号：配合 monthly-plan 一起用
```

## 关键规则

```
规则1：规划表是我自己的，TAPD是团队的，两个要对照看
规则2：组长直接写的需求 → 记录但不要追
规则3：迭代版本被删 → 先记录状态，再问原因
规则4：红标任务（上线迭代）→ 单独列，月底总结要覆盖
规则5：被跳过的需求 → 不是你的锅，但要记录在案
```
