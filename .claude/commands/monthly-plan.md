# 月度关键任务规划 Skill（monthly-plan）

## 使用场景

当用户说"规划下个月任务"、"关键任务规划"、"15号/31号上线安排"时调用。
每月25号前必须执行，用于提前规划下月的关键任务。

---

## 核心原则

```
1. 关键任务（15号/31号）需提前1个月开始准备
2. PRD评审需在计划上线前2周完成
3. 开发周期需预留buffer（实际 × 1.5）
4. 测试验收需在上线前1周启动
5. 风险项需提前识别并上报
```

---

## 执行步骤

### Step 1：确认当前月份和关键节点

```
输入：当前月份 [月]，关键任务节点 [15号/31号]

确认：
- 15号上线：哪些功能？
- 31号上线：哪些功能？
- 有哪些风险？
```

### Step 2：梳理关键任务清单

```
针对每个关键任务，确认：
1. PRD 是否已完成？ → 未完成需立即启动
2. PRD 是否已评审？ → 未评审需安排评审时间
3. 开发是否已排期？ → 未排期需找研发确认
4. 测试计划是否已定？ → 需确认测试资源
5. 风险项有哪些？ → 提前预警
```

### Step 3：生成月度规划表

```markdown
# [月] 月关键任务规划

## 一、关键任务节点

### 15号上线
| 功能 | PRD状态 | 开发状态 | 测试计划 | 风险 | 负责人 |
|------|--------|---------|---------|------|--------|
| [功能1] | ✅已完成 | 开发中 | 待安排 | 无 | @PM |

### 31号上线
| 功能 | PRD状态 | 开发状态 | 测试计划 | 风险 | 负责人 |
|------|--------|---------|---------|------|--------|
| [功能2] | ⏳评审中 | 待排期 | 待安排 | 可能延期 | @PM |

## 二、风险预警

| 风险 | 影响 | 应对方案 | 上报时间 |
|------|------|---------|---------|
| [风险1] | 可能导致31号延期 | [方案] | 提前1周上报 |

## 三、待确认事项

| # | 事项 | 确认人 | 确认时间 |
|---|------|--------|---------|
| 1 | [事项] | [@人] | [日期] |
```

### Step 4：输出到TAPD

```
将规划表同步到 TAPD，标记"关键任务"标签
通知相关人（研发/测试/业务）确认时间节点
```

---

## 输出格式

每次规划后输出：
1. 月度关键任务规划表（Markdown）
2. TAPD任务更新建议
3. 待确认事项清单
4. 风险预警（如有）

---

## 触发时机

```
自动触发：每月25号
手动触发：用户说"规划下个月"、"关键任务"
```

---

## 与其他 Skill 的配合

```
每月25号 → monthly-plan（生成下月规划）
          ↓
PRD未完成的 → prd-write / requirements-analysis
          ↓
开发未排期的 → 找研发确认排期
          ↓
测试资源不足 → 上报风险
          ↓
确认后 → git-push（推送规划到GitHub）
```
