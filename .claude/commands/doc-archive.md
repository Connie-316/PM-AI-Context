# 文档归档 Skill（doc-archive）

## 使用场景
当功能上线后，需要将 PRD 中的内容归档到 context/ 知识库时调用。
也可在用户说"归档"、"上线了"、"归档到知识库"时触发。

**触发条件（自动触发，不需要用户说）：**
1. PRD 文件中标注的状态变为"已完成"
2. 功能已在生产环境上线
3. 用户说"归档"

---

## 归档规则

### 文档生命周期

```
drafts/      →  workspace/prd/  →  context/
草稿             进行中             已上线
   ↓                ↓                  ↓
灵感/想法       PRD撰写中          功能上线后归档
```

### 归档原则

- **PRD 上线后**：从 `workspace/prd/` 移动到 `context/02_功能模块/`
- **UI 规范上线后**：从 Skill 文件夹更新到 `context/` 对应位置
- **不归档**：drafts/ 里的草稿不归档（那是还没想清楚的内容）
- **命名规范**：`[功能名称].md`，去掉日期后缀

---

## 执行步骤

### Step 1：识别待归档 PRD

扫描 `workspace/prd/` 目录，找出所有 PRD 文件。

### Step 2：确认归档条件

对于每个 PRD 检查：
- [ ] 功能是否已上线（状态为"已完成"或"已上线"）
- [ ] 验收标准是否已达成
- [ ] 是否需要归档（不是草稿）

### Step 3：从 PRD 提取归档内容

从 PRD 中提取以下内容写入 `context/02_功能模块/`：

```
## 功能名称

### 功能描述
[从 PRD 的"背景与目标"中提取，删除"为什么要做"部分，只保留"是什么、做什么"]

### 核心规则
[从 PRD 的"功能需求"中提取，保留规则和异常情况]

### 配置字段
[从 PRD 的"数据结构"中提取，只保留表名和字段]

### 状态说明
[从 PRD 的"功能需求"中提取，展示有哪些状态、颜色]
```

### Step 4：更新已上线功能列表

更新 `context/02_功能模块/已上线功能列表.md`：

```markdown
## [端]

### 功能名称
上线时间：[日期] | PRD：workspace/prd/[功能名称]_[日期].md
```

### Step 5：删除或保留原 PRD

- **保留**：原 PRD 保留在 `workspace/prd/`（标记为"已归档"）
- **移动**：如需移动，在 `workspace/prd/` 下建 `archive/` 子目录移入

---

## 归档后更新索引

归档完成后，自动更新：

1. `context/02_功能模块/CLAUDE.md` — 新增功能索引
2. `workspace/prd/CLAUDE.md` — 标注该 PRD 为"已归档"
3. `CLAUDE.md` — 如有必要更新
4. `.claude/SESSION_REPLAY.md` — 无需更新（这是工作记录，不关归档的事）

---

## 输出格式

```markdown
## 归档报告

**功能**：[功能名称]
**归档时间**：[时间]
**原 PRD**：workspace/prd/[功能名称]_[日期].md

### 归档内容

| 归档位置 | 内容 |
|---------|------|
| context/02_功能模块/[功能名称].md | 功能描述、规则、字段 |
| context/02_功能模块/已上线功能列表.md | 更新上线记录 |

### 索引更新

- ✅ 更新 context/02_功能模块/CLAUDE.md
- ✅ 更新 context/02_功能模块/已上线功能列表.md
- ✅ 原 PRD 保留在 workspace/prd/（标注"已归档"）

---

## 自动触发

以下情况**自动提醒归档**（用户确认后才执行）：
1. 用户说"功能上线了"
2. PRD 文件中状态标记为"已完成"

---

## 与其他 Skill 的配合

```
PRD 撰写完成 → prd-write
    ↓
审查 → doc-review
    ↓
功能上线 → doc-archive（提醒用户）
    ↓
归档完成后 → git-push（自动推送）
```
