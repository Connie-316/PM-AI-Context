# 周复盘 Skill（weekly-review）

## 使用场景

当用户说"周复盘"、"本周总结"、"每周复盘"时调用。
每周五下班前执行。

---

## 核心原则

```
1. 复盘是为了发现问题，不是自我感动
2. 每个问题都要有后续行动，否则不记录
3. 复盘要客观，不美化，不推卸
4. 复盘结果要让相关人知道（团队/领导）
```

---

## 执行步骤

### Step 1：收集本周数据

```
从 TAPD 导出：
- 本周完成任务列表
- 本周新增需求列表
- 本周变更需求列表
- 关键任务进度

从 需求池管理.md 导出：
- 本周新进需求 [X] 个
- 本周完成需求 [X] 个
- 本周取消需求 [X] 个

从 PRD 状态追踪 导出：
- 进行中 PRD [X] 个
- 已上线功能 [X] 个

---

## 与其他 Skill 的配合

```
weekly-review 生成复盘报告
    ↓
如有问题需要改流程 → 找组长沟通
    ↓
如有新需求 → 登记到 需求池管理.md
    ↓
如有PRD更新 → doc-version 保存版本
    ↓
确认后 → git-push
```

## 自动 Evolve 检查（知识飞轮）

**每次周复盘必须执行，不只写复盘报告：**

### 检查清单

```
□ drafts/ 有文件超过7天了吗？
   → 如果有，标注："[文件名] 在 drafts/ 超过7天，考虑晋升到 workspace/ 或删除"

□ workspace/prd/ 有功能已上线2周了吗？
   → 如果有，标注："[功能名] 已上线2周+，建议归档到 context/02_功能模块/"

□ context/ 有文件超过90天没更新了吗？
   → 如果有，标注："[文件名] 90天未动，问一下相关人是否还相关"
```
