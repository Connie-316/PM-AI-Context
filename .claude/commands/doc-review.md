# 文档审查 Skill（doc-review）

## 使用场景
当用户说"检查文档"、"审阅PRD"、"检查一致性"、"去掉冗余"、"修正错误"时调用。
每次文档修改后**自动触发**，不需要用户说。

---

## 核心原则

**每次修改后必须：**
1. 给出修改前后对比（精确到行）
2. 指出调整了哪些内容
3. 补充思考（修改背后的原因、潜在影响、可优化点）
4. 全面建议（超越"纠错"的范围，从更高视角给建议）

---

## 执行步骤

### Step 1：读取原文件

读取修改前的原文件内容（如有备份）。

### Step 2：读取新文件

读取修改后的文件内容。

### Step 3：逐项对比分析

**对比维度（6项，每项必须覆盖）：**

#### A. 内容调整对比（必须列出）

```
### 内容调整对比

| 位置 | 修改前 | 修改后 | 说明 |
|------|--------|--------|------|
| 3.1.2 颜色规范 | #3370FF | #1664FF | 统一主色，与截图一致 |
| 3.2 布局规范 | 缺少顶部导航描述 | 新增 56px #1A1A1A | 根据截图补充 |
| ... | ... | ... | ... |
```

#### B. 新增内容

```
### 新增内容
- 新增了 XX 内容，原因：...
- 新增了 XX 内容，原因：...
```

#### C. 删除内容

```
### 删除内容
- 删除了 XX 内容，原因：冗余/错误/重复
- 删除了 XX 内容，原因：...
```

#### D. 补充思考（超越纠错）

```
### 补充思考

1. **修改背后的原因**
   - 为什么这个颜色从 #3370FF 改到 #1664FF —— 因为截图里商家端主色是 #1664FF
   - 为什么加了这个规范 —— 因为截图里有这个 UI 但之前没记录

2. **潜在影响**
   - 这个修改可能影响：使用旧版 #3370FF 的地方需要同步更新
   - 建议：检查其他文档是否有引用旧值

3. **相关优化建议**
   - ui-spec-supplier 和 ui-spec-platform 引用了 ui-spec-merchant 的全部内容
     但两者主色目前还是 #3370FF，需要确认是否统一为 #1664FF
   - 建议给这三个 Skill 建立一个共享的"主色变量"，避免每个文件单独维护颜色值
```

#### E. 一致性全面检查

```
### 一致性检查

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 各端主色是否统一 | ⚠️ 需确认 | PC后台 #1664FF，小程序 #3370FF（历史原因） |
| 状态标签颜色是否一致 | ✅ 通过 | 五色系统在所有端统一 |
| 术语表与 PRD 描述是否一致 | ✅ 通过 | "商家"在所有文档中描述一致 |
| 文件引用路径是否正确 | ✅ 通过 | - |
| 表格格式是否统一 | ✅ 通过 | - |
```

#### F. 冗余和错误检查

```
### 冗余和错误

**冗余：**
- [ ] ui-spec-merchant.md 中"商家端、供应商端、平台端共用"这段文字在 ui-spec-supplier.md 和 ui-spec-platform.md 中重复出现，建议在一个文件里写一次，其他引用

**错误：**
- [ ] ui-spec-portal.md 中"搜索框圆角 18px"在 ui-demo-preview.html 中未体现，需补

**可优化：**
- [ ] ui-spec-miniprogram.md 引用了本地 HTML 文件路径，其他人到本机打开会失效，建议把规范内容直接写在 Skill 文件里（已做）
```

```markdown
## 修改操作（需用户确认）

如需执行修改，回复：
- **"修改"** → 执行所有修改，生成新文件
- **"修改第X项"** → 只修改指定项
- **"跳过"** → 保持原样，不修改
```

---

## 自动触发规则

**以下情况自动调用 doc-review，不需要用户说：**
1. 任何 Skill 生成文档后
2. 用户发送截图让我分析 UI 后
3. 用户说"帮我改一下这个文档"后
4. 每次会话开始时，若有待审文件

---

## 与其他 Skill 的配合

```
文档修改完成
    ↓
doc-review 自动审查
    ↓
输出：对比 + 补充思考 + 全面建议
    ↓
用户确认要修改哪些
    ↓
生成新文件（加 _v2 后缀）供确认
    ↓
确认后覆盖原文件
    ↓
归档或推送
```
